# Step 147: audio-decode Local Format Expansion

Date: 2026-07-01

## Plan

1. Use `audio-decode` as the single in-app local decode dependency and remove Coral's direct `@wasm-audio-decoders/flac` dependency.
2. Expand local import/playback candidates to the formats provided by `audio-decode`: MP3, WAV, OGG/OGA Vorbis, FLAC, Opus, M4A/AAC/ALAC, QOA, AIFF, CAF, WebM, AMR, and WMA.
3. Remove the Electron-native local file playback branch and route all supported local/downloaded files through `audio-decode`, wrap PCM as a WAV Blob, and reuse the existing decoded-audio runtime.
4. Implement DSF/DFF/SACD playback through a bundled FFmpeg binary behind the external decoder boundary — no user configuration required.

## Implemented

- Added `internalAudioDecodeExtensions` to the shared playback capability contract.
- Updated local audio defaults to include the full `audio-decode` extension set.
- Replaced the FLAC-specific runtime with `audio-decode` in `localAudioDecodeService`.
- Disabled direct native local-file playback in `musicUrlResolver`; local and downloaded files now decode through `audio-decode`.
- External decoder WAV outputs are decoded back through `audio-decode`, keeping playback on the same AudioBuffer path as other local files.
- Removed the app's direct `@wasm-audio-decoders/flac` dependency and the Vite prebundle exclusion for it. The package may still appear transitively through `audio-decode`'s FLAC subpackage.
- Updated playback capability smoke coverage and the long-running migration plan.

## audio-decode Codec Loading Fix

Three codecs (`@audio/decode-aac`, `@audio/decode-amr`, `@audio/decode-wma`) load
their WASM via `createRequire(import.meta.url)('./src/xxx.wasm.cjs')`. When Vite
SSR bundles them to CJS, `import.meta.url` is rewritten to
`require("url").pathToFileURL(__filename).href` and the relative `./src/xxx.wasm.cjs`
path no longer resolves under `dist/`.

Fix: added the three packages to `external` in `build-config/vite/main.config.ts`
so they stay as runtime `import()` calls and preserve ESM `import.meta.url`.
Also added the three `node_modules/@audio/decode-*` paths to the `files` whitelist
in `build-config/build-pack.js` so they ship in the package.

Verified post-build that all 13 codec entries in `audio-decode` load successfully.

## External Decoder: Bundled FFmpeg (replaces JustDSD)

### Background

Step 147 originally shipped DSF/DFF/SACD playback via JustDSD (a Java-based DSD
decoder) with a bundled JRE generated by `jlink`. This required ~40 MB of
platform-specific JRE assets, Java runtime management, and exposed a fragile
classpath/JAR resolution path that repeatedly broke in packaged builds.

The JustDSD integration has been **fully removed**. DFF/DSF/ISO/SACD files now
decode through a bundled `ffmpeg-static` binary — zero user configuration, zero
external runtime dependencies.

### Why FFmpeg binary over FFmpeg.wasm

A detailed evaluation compared FFmpeg native binary vs FFmpeg.wasm for the DSD
use case (200–500 MB files, 176.4 kHz sample rate):

| Dimension | FFmpeg binary | FFmpeg.wasm |
|-----------|--------------|-------------|
| DSD decoders | built-in (dsd_lsbf/msbf + planar, dst) | only in full ~25 MB build; pruned builds lose DSD |
| Performance | native speed, multi-core + SIMD | 0.04–0.08x native (12–25x slower) |
| Memory | process memory, no cap | WASM 32–64 MB cap → DSD files OOM |
| Streaming | spawn → stdout PCM, natural fit | VFS model, must load whole file to memory |
| Electron | mature (asar + spawn) | requires COOP/COEP headers, breaks cross-origin |

FFmpeg binary wins decisively for real-time DSD playback in a desktop app.

### Bundled assets

- `node_modules/ffmpeg-static` — the `ffmpeg-static` npm package ships a
  per-platform static FFmpeg binary (FFmpeg 6.0, ~70–120 MB). Resolved at runtime
  via `require('ffmpeg-static')` which returns the absolute path to the binary.
- Included in the electron-builder `files` whitelist in `build-config/build-pack.js`
  so it ships inside the app. With `asar: false` the binary is directly executable
  without `asarUnpack`.

### Runtime resolution

`src/main/modules/winMain/externalDecoderRuntime.ts` resolves the FFmpeg path via
`resolveBundledFfmpegPath()`:

```ts
const ffmpegStatic = require('ffmpeg-static') as string;
return ffmpegStatic.replace('app.asar', 'app.asar.unpacked');
```

The `.replace('app.asar', 'app.asar.unpacked')` is a safety net for asar builds;
in the current `asar: false` configuration it is a no-op. The resolved path is
passed to `child_process.spawn` with args:

```text
ffmpeg -y -hide_banner -loglevel error -i <input> -vn -acodec pcm_s16le -ar 44100 <output.wav>
```

The output WAV is then decoded back through `audio-decode` in the renderer,
keeping all local files on the same AudioBuffer playback path.

### Simplified external decoder contract

The external decoder API was simplified to remove all provider/path configuration:

- `ExternalDecoderProvider` type removed — FFmpeg is the only backend.
- `ExternalDecoderTranscodeParams` reduced to `{ inputPath, output, timeoutMs }`.
- `ExternalDecoderProbeParams` reduced to `{ extensions }`.
- Settings removed: `provider`, `executablePath`, `javaPath`, `justDsdJarPath`,
  `pluginDirs`. Only `enabled`, `preferredOutput`, `extensions`, `timeoutMs` remain.
- `foobar2000DecoderPluginHints` removed (Foobar2000 provider dropped).
- UI simplified to a single enable switch + output format + timeout + extensions
  + probe button. No path inputs, no provider radio, no plugin directory textarea.

### Removed JustDSD assets

- `resources/justdsd/` directory (jdsd-nodep.jar, CoralJustDsdWavExporter.class/.java,
  README.md, jre/) — deleted entirely.
- `build-config/prepare-justdsd.js` — deleted.
- `build-config/pack.js` — `require('./prepare-justdsd')` call removed.
- `build-config/build-pack.js` — `{ from: './resources/justdsd', to: 'justdsd' }`
  extraResources entry removed.
- `.gitignore` — `resources/justdsd/jre/` and `runtime/` ignore rules removed.
